# jj-lib integration guide (for `jjscratch`)

In-process integration guide for reading a Jujutsu repository via the `jj-lib`
crate (no shelling out to the `jj` CLI).

- **Pinned version: `jj-lib` 0.42.0** (latest on crates.io as of 2026-06-04).
- **Toolchain target: rustc 1.95.0 / cargo 1.95.0.** jj-lib 0.42.0 declares
  `rust-version = "1.89"` and `edition = "2024"`, so 1.95 is comfortably above
  the MSRV. edition 2024 in jj-lib is fine for a 2021/2024-edition consumer.

Verification: every type/function name below was read from the v0.42.0 source
tag (`https://github.com/jj-vcs/jj/tree/v0.42.0/lib/src`) and cross-checked
against docs.rs where noted. Confidence levels are called out per section.

Top-level docs: https://docs.rs/jj-lib/0.42.0/jj_lib/

---

## 1. Version, dependencies, and build prerequisites

### Cargo.toml

```toml
[dependencies]
# default features include "git" (pulls in the optional `gix` backend, which
# is what you need to read a normal jj-on-git repo). Keep defaults on.
jj-lib = "0.42.0"

# Needed to drive jj-lib's async APIs from sync code (see notes below). jj-lib
# itself re-exports nothing for this; pick your own executor. `pollster` is the
# lightest option and is already a transitive dep of jj-lib.
pollster = "0.4"
futures = "0.3"
```

`jj-lib` feature flags (from `lib/Cargo.toml`):

- `default = ["git"]` — enables `git = ["dep:gix"]`. **Keep this on**; a jj repo
  colocated with or backed by git needs the `gix` (gitoxide) backend.
- `watchman` — filesystem watcher; not needed for read-only.
- `testing` — test helpers; not needed.

Note: jj-lib uses **`gix` (gitoxide), not `git2`/libgit2** for git access, so
there is no C linkage / system libgit2 requirement from the git backend.

### protoc — NOT required (biggest assumed risk, debunked)

This is the single most important finding for build reliability.

jj-lib uses `prost` for protobuf, **but it does NOT compile `.proto` files at
build time.** Verified facts for v0.42.0:

- `lib/Cargo.toml` has **no `[build-dependencies]` and no `build.rs`**. `prost`
  is a normal runtime dependency; `prost-build` is *not* a dependency of
  `jj-lib`.
- The generated Rust is **checked into the source tree**: e.g.
  `lib/src/protos/git_store.rs`, `default_index.rs`, `simple_op_store.rs`, etc.,
  each `include!`-d from `lib/src/protos/mod.rs` (header comment:
  `// This file is @generated by prost-build.`).
- Regeneration is done out-of-band by a separate dev-only binary crate,
  `lib/gen-protos` (which is the *only* thing that depends on `prost-build`),
  and is not part of the published `jj-lib` crate's build.

**Conclusion: building `jj-lib` as a dependency does NOT require `protoc` on the
system, and sets no `PROTOC` env var.** `protobuf-src` / vendored protoc is not
needed. (The historical concern about jj needing system `protoc` no longer
applies to recent releases — confidence: high, confirmed against the v0.42.0
manifest and tree listing.)

### Other build notes

- No special env vars are required.
- Transitive native deps are minimal (blake2, gix). On Unix jj-lib pulls
  `rustix`; on Windows `winreg`/`same-file`. No system protobuf/libgit2.

## 2. Loading a repository

Types: `Workspace`, `DefaultWorkspaceLoader` / `WorkspaceLoaderFactory`,
`RepoLoader`, `Repo` (trait), `ReadonlyRepo`, `UserSettings`, `StoreFactories`,
`StackedConfig`.

Key facts (verified against `lib/src/workspace.rs`, `repo.rs`, `settings.rs`,
`config.rs`):

- `Workspace::load(user_settings, workspace_path, store_factories,
  working_copy_factories) -> Result<Workspace, WorkspaceLoadError>`. The
  `workspace_path` is the working-copy root (the directory containing `.jj`).
- `Workspace` gives you `repo_loader() -> &RepoLoader`, `settings()`,
  `workspace_name() -> &WorkspaceName`, `working_copy()`.
- **The repo is loaded from the loader, and it is `async`:**
  `RepoLoader::load_at_head() -> impl Future<Output = Result<Arc<ReadonlyRepo>,
  RepoLoaderError>>`. (Also `load_at(&Operation)`.)
- `UserSettings::from_config(StackedConfig) -> Result<UserSettings, _>`;
  build the config with `StackedConfig::with_defaults()` for the built-in
  default variables. (For a real app you would also layer in the user's
  `~/.config/jj/config.toml`, but defaults are enough to open and read a repo.)
- `StoreFactories::default()` and `default_working_copy_factories()` provide the
  standard git/simple backends and the local working copy.
- The working-copy commit id is read from the `View`:
  `repo.view().get_wc_commit_id(workspace_name) -> Option<&CommitId>` (or
  `wc_commit_ids()` for the full map). `View::is_wc_commit_id(&CommitId) -> bool`.

```rust
use std::path::Path;
use std::sync::Arc;

use jj_lib::config::StackedConfig;
use jj_lib::repo::{ReadonlyRepo, Repo, StoreFactories};
use jj_lib::settings::UserSettings;
use jj_lib::workspace::{default_working_copy_factories, Workspace};

/// Open the jj workspace whose working-copy root is `path`, returning the repo
/// at the head operation plus the working-copy commit id for this workspace.
pub fn open_repo(
    path: &Path,
) -> anyhow::Result<(Workspace, Arc<ReadonlyRepo>, Option<jj_lib::backend::CommitId>)> {
    // 1. Settings. with_defaults() supplies jj-lib's built-in config variables.
    let config = StackedConfig::with_defaults();
    let settings = UserSettings::from_config(config)?;

    // 2. Load the workspace (.jj/ under `path`).
    let workspace = Workspace::load(
        &settings,
        path,
        &StoreFactories::default(),
        &default_working_copy_factories(),
    )?;

    // 3. Load the repo at the current head operation. This is async; block on it.
    let repo: Arc<ReadonlyRepo> =
        pollster::block_on(workspace.repo_loader().load_at_head())?;

    // 4. Working-copy commit id for this workspace.
    let wc_name = workspace.workspace_name().to_owned();
    let wc_commit_id = repo.view().get_wc_commit_id(&wc_name).cloned();

    Ok((workspace, repo, wc_commit_id))
}
```

Confidence: high on the type/method names and the async-ness of
`load_at_head`. The exact `anyhow` error conversions are illustrative — the real
error types are `WorkspaceLoadError` and `RepoLoaderError`.

Docs: https://docs.rs/jj-lib/0.42.0/jj_lib/workspace/struct.Workspace.html ,
https://docs.rs/jj-lib/0.42.0/jj_lib/repo/struct.RepoLoader.html

## 3. Commit graph / log (DAG enumeration)

Types: `RevsetExpression` (state-parameterized: `UserRevsetExpression` vs
`ResolvedRevsetExpression`), the `Revset` trait, `GraphNode` / `GraphEdge` from
`jj_lib::graph`, `Commit`, `CommitId`, `Store`.

Critical API facts (verified against `lib/src/revset.rs`, `graph.rs`):

- The `Revset` trait is **stream-based** in 0.42.0 (confirmed on docs.rs). Its
  required methods are:
  - `stream() -> LocalBoxStream<Result<CommitId, RevsetEvaluationError>>`
    (topological, children before parents),
  - `commit_change_ids() -> LocalBoxStream<Result<(CommitId, ChangeId), _>>`,
  - `stream_graph() -> LocalBoxStream<Result<GraphNode<CommitId>, _>>`
    (**this is the one you want for a DAG view** — it yields each node plus its
    edges),
  - `is_empty()`, `count_estimate()`, `containing_fn()`.
  There is no synchronous `iter()` anymore; drive the stream with
  `pollster::block_on` + `futures::StreamExt`.
- `GraphNode<N, ID = N> = (N, Vec<GraphEdge<ID>>)` (a tuple). `GraphEdge` has
  `pub target: ID` and `pub edge_type: GraphEdgeType` (`Direct` / `Indirect` /
  `Missing`; helpers `is_direct()`, `is_indirect()`, `is_missing()`). Indirect
  edges happen when intermediate commits are filtered out of the revset.
- Build the expression **programmatically** via the builder methods, which avoids
  the string parser entirely. Constructors like `RevsetExpression::all()`,
  `none()`, `root()`, `commits(vec)`, and combinators `.ancestors()`,
  `.heads()`, `.parents()`, `.union()`, etc. are generic over the expression
  state, so you can construct them directly as a `ResolvedRevsetExpression` (no
  symbol resolution step) when the expression contains no symbols.
- Evaluate with `ResolvedRevsetExpression::evaluate(self: Arc<Self>, repo: &dyn
  Repo) -> Result<Box<dyn Revset>, RevsetEvaluationError>`.
- Get a `Commit` from a `CommitId`: `repo.store().get_commit(&id)` (sync) or
  `get_commit_async`. `RevsetStreamExt::commits(store)` adapts a
  `Stream<CommitId>` into a `Stream<Commit>`.

> The CLI's default `jj log` revset is the alias
> `present(@) | ancestors(immutable_heads().., 2) | trunk()` (from
> `cli/src/config/revsets.toml`). That relies on the CLI's revset *aliases*
> (`immutable_heads()`, `trunk()`), which are **not** built into jj-lib — they
> are CLI config. For an app, prefer building the expression yourself. A close
> equivalent to "all visible commits" is `RevsetExpression::all()`. See §4 for
> reconstructing `immutable()` from lib primitives.

```rust
use std::sync::Arc;

use futures::StreamExt as _;
use jj_lib::graph::GraphEdgeType;
use jj_lib::repo::{ReadonlyRepo, Repo};
use jj_lib::revset::{ResolvedRevsetExpression, RevsetExpression};

/// Stream every visible commit with its outgoing (parent) graph edges, in
/// topological order (children before parents).
pub fn for_each_graph_node(repo: &Arc<ReadonlyRepo>) -> Result<(), Box<dyn std::error::Error>> {
    // `all()` is generic over expression state; annotate it as a resolved
    // expression so no symbol resolution is needed.
    let expr: Arc<ResolvedRevsetExpression> = RevsetExpression::all();

    let revset = expr.evaluate(repo.as_ref())?;

    pollster::block_on(async {
        let mut stream = revset.stream_graph();
        while let Some(node) = stream.next().await {
            let (commit_id, edges) = node?;
            let commit = repo.store().get_commit(&commit_id)?; // -> jj_lib::commit::Commit
            for edge in &edges {
                let _kind = match edge.edge_type {
                    GraphEdgeType::Direct => "direct",
                    GraphEdgeType::Indirect => "indirect", // intermediate nodes elided
                    GraphEdgeType::Missing => "missing",
                };
                let _parent_id = &edge.target; // CommitId of the parent node
            }
            // ... feed (commit, edges) into your DAG renderer ...
            let _ = commit;
        }
        Ok::<_, Box<dyn std::error::Error>>(())
    })
}
```

For just the parent ids of a single commit (no graph topology needed), skip the
revset entirely: `commit.parent_ids() -> &[CommitId]` (see §4).

Confidence: high on `Revset` being stream-based and on `stream_graph` /
`GraphNode` / `GraphEdge` shapes (confirmed on docs.rs + source). Medium on the
exact "build resolved expression directly" ergonomics — if the type annotation
on `RevsetExpression::all()` does not infer cleanly, the fully general path is:
build a `UserRevsetExpression`, then
`expr.resolve_user_expression(repo, &SymbolResolver::new(repo, &[]))?` to get a
`ResolvedRevsetExpression`, then `.evaluate(repo)`.

Docs: https://docs.rs/jj-lib/0.42.0/jj_lib/revset/trait.Revset.html ,
https://docs.rs/jj-lib/0.42.0/jj_lib/revset/enum.RevsetExpression.html ,
https://docs.rs/jj-lib/0.42.0/jj_lib/graph/index.html

## 4. Commit metadata, immutability, working-copy, divergence, bookmarks

### Basic metadata (all verified in `lib/src/commit.rs`, `backend.rs`)

`Commit` accessors:

- `commit.id() -> &CommitId`
- `commit.change_id() -> &ChangeId`
- `commit.description() -> &str`
- `commit.author() -> &Signature`, `commit.committer() -> &Signature`
- `commit.parent_ids() -> &[CommitId]`
- `commit.tree() -> MergedTree`, `commit.tree_ids() -> &Merge<TreeId>`
- `commit.has_conflict() -> bool`

`Signature` (a plain struct, public fields): `name: String`, `email: String`,
`timestamp: Timestamp`. `Timestamp` has `timestamp: MillisSinceEpoch(i64)` and
`tz_offset: i32` (minutes); convert with `timestamp.to_datetime() ->
Result<chrono::DateTime<FixedOffset>, _>`.

```rust
use jj_lib::commit::Commit;

fn describe(commit: &Commit) {
    let author = commit.author();
    let when = author.timestamp.to_datetime().ok();
    println!(
        "commit {} change {} by {} <{}> at {:?}\n{}",
        commit.id().hex(),
        commit.change_id().hex(),
        author.name,
        author.email,
        when,
        commit.description(),
    );
    for parent in commit.parent_ids() {
        println!("  parent {}", parent.hex());
    }
}
```

### Is it the working-copy commit?

`repo.view().is_wc_commit_id(commit.id()) -> bool`, or compare against
`view().get_wc_commit_id(workspace_name)`. For multi-workspace repos,
`view().workspaces_for_wc_commit_id(commit.id()) -> Vec<WorkspaceNameBuf>` tells
you which workspaces point at it.

### Is it immutable?

There is **no built-in `immutable()` in jj-lib** — `immutable_heads()` /
`immutable()` / `trunk()` are CLI revset *aliases* (config), shown here from
`cli/src/config/revsets.toml`:

```text
builtin_immutable_heads() = trunk() | tags() | untracked_remote_bookmarks()
immutable_heads()         = builtin_immutable_heads()   # user-overridable
immutable()               = ::(immutable_heads() | root())
mutable()                 = ~immutable()
```

For an app you reconstruct this from lib primitives. The robust pattern: build
a `RevsetExpression` for your chosen immutable heads (e.g. tags + trunk +
untracked remote bookmarks, or whatever your app exposes as config), take
`.ancestors()` (which is `::`), union with `RevsetExpression::root()`, evaluate
it, and then use the evaluated revset's `containing_fn()`:

```rust
// expr_immutable: Arc<ResolvedRevsetExpression> describing ::(<heads> | root())
let immutable = expr_immutable.evaluate(repo.as_ref())?;
let is_immutable = immutable.containing_fn(); // Box<dyn Fn(&CommitId) -> Result<bool, _>>
let immutable_here = is_immutable(commit.id())?;
```

`containing_fn()` is the intended way to test membership of many commits against
one revset efficiently. Confidence: high on `containing_fn` existing and its
signature; medium on the exact heads you choose (that is product policy, mirror
the alias above for parity with `jj`).

### Is it divergent?

Divergent = a visible change id shared by more than one visible commit. jj-lib
exposes `RevsetExpression::divergent()` directly (an `AsFilter(Divergent)`
predicate). Evaluate `RevsetExpression::divergent()` (intersected with your
visible set if desired) and use `containing_fn()` as above, or collect its
commit ids into a set. Alternatively, the `Repo` trait has
`resolve_change_id(&ChangeId) -> Result<Option<ResolvedChangeTargets>>` to find
all commits for a change id and check whether there is more than one.

### Bookmarks / refs pointing at a commit

From `lib/src/view.rs` + `op_store.rs`:

- Local bookmarks at a commit:
  `view.local_bookmarks_for_commit(commit_id) -> impl Iterator<Item = (&RefName,
  &RefTarget)>`.
- All local bookmarks: `view.local_bookmarks()`; remote:
  `view.all_remote_bookmarks() -> impl Iterator<Item = (RemoteRefSymbol,
  &RemoteRef)>`, `view.remote_bookmarks(...)`.
- Tags: `view.local_tags()`, `view.all_remote_tags()`. Git refs:
  `view.git_refs()`, `view.git_head()`.
- `RefTarget`: `added_ids() -> impl Iterator<Item = &CommitId>` (the ids it
  resolves to; a conflicted ref can have several), `as_normal() ->
  Option<&CommitId>` (the single id when unconflicted), `is_absent()`.

```rust
use jj_lib::repo::Repo;

fn bookmarks_at(repo: &dyn Repo, commit_id: &jj_lib::backend::CommitId) -> Vec<String> {
    repo.view()
        .local_bookmarks_for_commit(commit_id)
        .map(|(name, _target)| name.as_str().to_owned())
        .collect()
}

// Remote bookmarks pointing at a commit (filter the full iterator):
fn remote_bookmarks_at(repo: &dyn Repo, commit_id: &jj_lib::backend::CommitId) -> Vec<String> {
    repo.view()
        .all_remote_bookmarks()
        .filter(|(_, remote_ref)| remote_ref.target.added_ids().any(|id| id == commit_id))
        .map(|(symbol, _)| symbol.to_string())
        .collect()
}
```

Confidence: high on `local_bookmarks_for_commit`, `local_bookmarks`,
`all_remote_bookmarks`, and `RefTarget::added_ids`. Medium on the exact field
access `remote_ref.target` — `RemoteRef` carries a `target: RefTarget` plus a
tracking state; confirm the field/method name on docs.rs for
`jj_lib::op_store::RemoteRef` if it does not compile.

Docs: https://docs.rs/jj-lib/0.42.0/jj_lib/view/struct.View.html ,
https://docs.rs/jj-lib/0.42.0/jj_lib/op_store/struct.RefTarget.html ,
https://docs.rs/jj-lib/0.42.0/jj_lib/commit/struct.Commit.html

## 5. Diffs (commit vs parent)

Types/modules: `MergedTree` (`merged_tree`), `Diff<T>` (`merge`),
`TreeDiffEntry`, `TreeDiffStream`, `MaterializedTreeValue` /
`MaterializedFileValue` (`conflicts`), `Store::read_file`, `Matcher`
(`matchers::EverythingMatcher`).

Verified facts:

- A commit's tree is `commit.tree() -> MergedTree`. To diff against the (first)
  parent: get `parent = store.get_commit(&commit.parent_ids()[0])?` then
  `parent.tree()`. (Merge commits have multiple parents; for a diff view you
  typically diff against parent 0, or auto-merge the parents — start with
  parent 0.)
- Diff is a **stream**:
  `parent_tree.diff_stream(&commit_tree, matcher: &dyn Matcher) ->
  TreeDiffStream` (= `BoxStream<TreeDiffEntry>`). Use
  `jj_lib::matchers::EverythingMatcher` to include all paths. There are also
  `diff_stream_with_copies(..., &CopyRecords)` and
  `diff_stream_with_copy_history(...)` variants if you want rename/copy
  tracking; get `CopyRecords` via `store.get_copy_records(...)`.
- `TreeDiffEntry { pub path: RepoPathBuf, pub values: BackendResult<Diff<MergedTreeValue>> }`.
  `Diff<T> { pub before: T, pub after: T }` (here `T = MergedTreeValue =
  Merge<Option<TreeValue>>`). `before` absent => added; `after` absent =>
  deleted.
- To turn a `MergedTreeValue` into bytes, use
  `jj_lib::conflicts::materialize_tree_value(store, path, value).await ->
  BackendResult<MaterializedTreeValue>`. For an ordinary file you get
  `MaterializedTreeValue::File(MaterializedFileValue { reader, .. })` and call
  `file.read_all(path).await -> BackendResult<Vec<u8>>`. (Or, when you already
  have a resolved `FileId`, read directly with `store.read_file(path, &id)`
  which returns an `AsyncRead`.)
- For unified-diff *text* / hunks, jj-lib's `jj_lib::diff` module provides the
  line-diff engine (`Diff`, `DiffHunk { kind, contents }`, `DiffHunkKind`,
  `DiffHunkIterator`). The CLI layers its unified-diff formatting on top of
  this; jj-lib gives you the before/after byte blobs and the hunk primitives,
  not a ready-made `--git` formatter. Easiest path for `jjscratch`: collect
  before/after blobs (below) and feed them to a unified-diff crate, or walk
  `jj_lib::diff::Diff::default_refinement(&[before, after]).hunks()`.

```rust
use futures::StreamExt as _;
use jj_lib::commit::Commit;
use jj_lib::conflicts::{materialize_tree_value, MaterializedTreeValue};
use jj_lib::matchers::EverythingMatcher;
use jj_lib::repo::ReadonlyRepo;
use std::sync::Arc;

/// Yield (path, before_bytes, after_bytes) for every changed file between a
/// commit and its first parent. Absent side => empty Vec (added/deleted).
pub fn changed_files(
    repo: &Arc<ReadonlyRepo>,
    commit: &Commit,
) -> Result<Vec<(String, Vec<u8>, Vec<u8>)>, Box<dyn std::error::Error>> {
    let store = repo.store().clone();
    let to_tree = commit.tree();
    let from_tree = match commit.parent_ids().first() {
        Some(pid) => store.get_commit(pid)?.tree(),
        None => store.empty_merged_tree(), // root commit: parent is the empty tree
    };

    pollster::block_on(async {
        let mut out = Vec::new();
        let mut stream = from_tree.diff_stream(&to_tree, &EverythingMatcher);
        while let Some(entry) = stream.next().await {
            let path = entry.path; // RepoPathBuf
            let diff = entry.values?; // Diff<MergedTreeValue> { before, after }

            let before = read_value(&store, &path, diff.before).await?;
            let after = read_value(&store, &path, diff.after).await?;
            out.push((path.as_internal_file_string().to_owned(), before, after));
        }
        Ok::<_, Box<dyn std::error::Error>>(out)
    })
}

async fn read_value(
    store: &Arc<jj_lib::store::Store>,
    path: &jj_lib::repo_path::RepoPath,
    value: jj_lib::backend::MergedTreeValue,
) -> Result<Vec<u8>, Box<dyn std::error::Error>> {
    match materialize_tree_value(store, path, value).await? {
        MaterializedTreeValue::Absent => Ok(Vec::new()),
        MaterializedTreeValue::File(mut f) => Ok(f.read_all(path).await?),
        // Symlinks, conflicts, submodules, trees: render as you see fit.
        _ => Ok(Vec::new()),
    }
}
```

Confidence: high on `diff_stream` + `TreeDiffEntry`/`Diff<T>` shapes,
`materialize_tree_value`, and `MaterializedFileValue::read_all` (all read from
source). Medium on the exact import paths of `MergedTreeValue` (re-exported from
both `backend` and `merge` as `Merge<Option<TreeValue>>`) and on
`Store::empty_merged_tree()` vs `empty_merged_tree_id()` — both exist in
`store.rs`; use the one returning a `MergedTree`. The `jj_lib::diff` hunk API
(`DiffHunk`, `DiffHunkKind`, `DiffHunkIterator`) is confirmed present but its
exact entry function for line diffs should be checked on docs.rs before relying
on it.

Docs: https://docs.rs/jj-lib/0.42.0/jj_lib/merged_tree/struct.MergedTree.html ,
https://docs.rs/jj-lib/0.42.0/jj_lib/conflicts/index.html ,

## Cross-cutting notes

- **Async everywhere.** Repo loading, revset streams, and diff/file reads are
  all `async`. jj-lib does not pick an executor for you. The lightest option is
  `pollster::block_on(...)` around each call (used throughout above). If you
  already have a Tokio runtime, you can `block_on` on it instead. The streams
  are `LocalBoxStream` / `BoxStream`, driven with `futures::StreamExt::next`.
- **Ids print with `.hex()`.** `CommitId`/`ChangeId` are byte ids;
  `id.hex() -> String` gives the display form.
- **`Repo` is a trait.** `ReadonlyRepo: Repo`. Most read APIs take `&dyn Repo`;
  pass `repo.as_ref()` from an `Arc<ReadonlyRepo>`.
- **Stability.** jj is pre-1.0 and explicitly experimental; the library API
  changes between minor releases (e.g. the `Revset` trait went stream-only).
  Pin `=0.42.0` and re-audit on upgrade.
